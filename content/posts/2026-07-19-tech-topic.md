## RAG 已死？还是刚学会走路

如果你经常刷技术社区，大概已经看过不下十篇「RAG is Dead」的标题了。这个说法每隔几个月就要被拎出来遛一圈，仿佛 Retriever-Augmented Generation 是什么过时的旧玩具。

但说实话，RAG 不是死了。它只是终于走出了"随便搜几段相关文本拼进去"的婴儿期。

### 纯 RAG 的软肋在哪

用过的人都懂——经典的 Naive RAG 流程就三步：用户问 → 向量检索找 top-k 片段 → 塞进 prompt 让 LLM 回答。

看起来流畅，但深坑不少：

- **语义鸿沟**：用户问的是"Q3 营收为什么下滑"，检索回来的是"2024年9月财务报告第27页"，向量相似度高但回答不了因果推理。
- **碎片化问题**：答案分散在三段不连续的文本里，LLM 被迫自己脑补中间缺失的逻辑链条。
- **信息孤岛**：每个 chunk 都是一座孤岛，实体之间的关系、事件的时间线、概念之间的层级结构，全被忽略了。

说白了，纯 RAG 擅长的是"找原文片段"，不擅长的是"把碎片拼成完整的推理链"。

### 知识图谱进来救场

解决这个问题的思路其实很朴素：如果文本碎片不够用，那就把碎片之间的"关系"也带上。

GraphRAG（Microsoft 2024 年提出）是最典型的代表——先对文档做实体提取和关系建图，查询时不是搜段落，而是在图上遍历找到相关的子图，再把子图序列化成文本喂给 LLM。

今年蚂蚁集团开源的 KAG（Knowledge-Augmented Generation）把这个思路又往前推了一步。它不光建图，还引入了**逻辑形式的规划器**——简单说就是先规划"这个问题需要查什么、按什么顺序查、中间结果怎么拼接"，再执行检索。

你给它一个复杂问题，它先拆成几步推理任务，每一步在图谱上查，结果缓存起来传给下一步，最后汇总输出。这已经不是"检索增强生成"了，更像是"推理增强生成"。

### 变化背后是什么

说穿了，2026 年的技术圈对 LLM 的期待变了。

两年前大家问的是"能不能答准"，现在问的是"能不能想清楚"。单个 QA 任务上 RAG 早就够用了，但一旦涉及到多跳推理、对比分析、时序因果，纯检索那一套就不太行了。

从这个角度看，RAG 的进化方向其实就是 LLM 应用从"问答工具"走向"推理工具"的自然映射：

```
Naive RAG → Agentic RAG → GraphRAG / KAG → ...
```

工具链在变，但核心矛盾一直没变——**如何让模型既利用外部知识，又能基于这些知识做真正的推理**。

### 一点个人看法

我不觉得 RAG 会消失，它最终会成为和 Agent、Tool-use 并列的"脚手架"之一。未来的 AI 应用大概率是混着来的：简单问题向量检索直接答，复杂问题上图谱推理，工具类问题调 API。

RAG 不是死了，它终于在学会走路了。

至于"RAG is Dead"这类标题 —— 嗯，流量密码嘛，懂的都懂。
